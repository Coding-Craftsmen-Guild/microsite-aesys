# Hide Children From Navigation — Design

**Date:** 2026-06-23
**Status:** Approved

## Problem

Pages can already be excluded from the top navigation individually via the
existing **Hide From Navigation** toggle on the `page` Settings tab
(`NavigationService` filters on `page.HideFromNavigation`). There is no way to
keep a page in the nav while suppressing *its children*.

Concrete need: the Blog Landing Page should appear in the navigation, but its
child blog articles should not be listed as a nav sub-menu.

## Goal

Add a second per-page boolean, **Hide Children From Navigation**, that — when
set on a page — removes that page's children (and their subtree) from the top
navigation while keeping the page itself visible. Both flags coexist on the
Settings tab; they are independent:

| `HideFromNavigation` | `HideChildrenFromNavigation` | Result |
|---|---|---|
| false | false | page shown, children shown (current default) |
| true  | —     | page (and its subtree) absent from nav |
| false | true  | page shown, children hidden |

The flag lives on the shared `page` base doctype, so every page type
(HomePage, StandardPage, BlogLandingPage, BlogPage) inherits it. To hide blog
articles, an editor sets the flag on the **Blog Landing Page** — the parent
that holds the article children.

## Changes

### 1. Schema — `Aesys.Core/Pages/Page/page.config`

Add one `Umbraco.TrueFalse` property to the existing **Settings** tab
(`<Tab Alias="settings">`), alongside the existing `hideFromNavigation`:

- `Alias`: `hideChildrenFromNavigation`
- `Name`: `Hide Children From Navigation`
- `Description`: `Exclude this page's child pages from the main navigation (the page itself stays).`
- `Definition`: `92897bc6-a5f3-4ffe-ae27-f2e7e33dda49` — reuses the same
  TrueFalse DataType as `hideFromNavigation`.
- `Key`: `44d10e6f-687d-42b2-b55d-3e56edaf7d90` — verified globally unique
  across `Aesys.Core/` and `Aesys.Web/uSync/` per the usync-author
  GUID-uniqueness rule.
- `Variations`: `Nothing` (matches `hideFromNavigation`).

The Settings tab is already declared on `page.config`, so no tab pre-flight
issue applies.

### 2. Behavior — `Aesys.Core/Services/NavigationService.cs`

In `BuildChildren`, after mapping each child `Page`, skip descending into its
own children when `page.HideChildrenFromNavigation` is set. The page node is
still emitted; only its `Children` collection is forced empty.

Currently:

```csharp
var children = includeGrandchildren
    ? BuildChildren(page.Key, current, includeGrandchildren: false)
    : [];
```

Becomes:

```csharp
var children = includeGrandchildren && !page.HideChildrenFromNavigation
    ? BuildChildren(page.Key, current, includeGrandchildren: false)
    : [];
```

The root HomePage's direct children are built in `GetTopNavigation` via
`BuildChildren(root.Key, …)`. To honour the flag on the HomePage root itself,
that call is likewise guarded with `root.HideChildrenFromNavigation`.

## Bootstrap ordering (required)

`ModelsMode` is `SourceCodeAuto`: the new `Page.HideChildrenFromNavigation`
member is generated by the running app, not by `dotnet build`. So C# that reads
it cannot compile until the app has run once. Order:

1. Edit `page.config` (schema change).
2. `mise run usync:bundle`.
3. Run the app via `mise run dev` so uSync imports the schema and ModelsBuilder
   regenerates `Aesys.Core/Generated/Page.generated.cs` with the new property.
4. Then edit `NavigationService.cs` to reference the new member; `dotnet watch`
   rebuilds green.

## Testing

Manual verification in the running app:

1. Backoffice → Page Settings tab shows **both** toggles.
2. Set **Hide Children From Navigation** on the Blog Landing Page; publish.
3. Header nav: Blog Landing Page link still renders; its article children no
   longer appear as a sub-menu.
4. Regression: a page with neither flag still shows page + children; a page with
   **Hide From Navigation** is still fully absent.

## Out of scope

- No change to the existing `HideFromNavigation` behavior.
- No UFM label / block-editor changes.
- No new DataType (the existing TrueFalse editor is reused).
